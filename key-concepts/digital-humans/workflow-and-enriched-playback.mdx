Some digital humans are driven by **workflow adherence** or **enriched playback** instead of classic intent / transcript LLM flows. The API applies extra rules when a test case is **workflow-tagged** or when **`enriched_playback`** is present.

## What counts as a “workflow” digital human

A test case is treated as workflow playback when its **tag** matches the platform rule: the **lowercased** tag string **contains `"workflow"`** (for example, “Workflow Adherence”). Tags are stored on the test case; on **create**, you set the single **`tag`** field on the request body.

The platform applies the same rule in workflow digital-human utilities (tag string is lowercased and checked for the substring `workflow`).

## `POST /v1/create-digital-human`

### A. Any create with non-empty `enriched_playback`

When `enriched_playback` is a **non-empty list**:

- Transcript LLM paths are skipped (`original_transcript` / `formatted_transcript` processing for intent).
- **`description`** and **`success_criteria`** are persisted as **empty strings** (even if you send values). **`speaks_first_config`** and optional **`formatted_transcript`** from the request are kept as provided.
- **`generate_objectives_and_intent_summary`** is **not** run when description is blank after create.

### B. Workflow-tagged create (tag matches the rule above)

**400** when (after trim) any of these hold:

- Non-empty **intent**, **success_criteria**, or **role_description**
- **`original_transcript`** or **`formatted_transcript`** is present
- **`enriched_playback`** is missing, not a list, or **empty** — workflow-tagged creates require a **non-empty list**

**`role_description`** is stored as **`null`** even if omitted (forced for workflow-tagged creates).

### C. `DigitalHumanRequestData` (create body)

- **`intent`** and **`success_criteria`** default to **`""`** and are optional for enriched-playback-first flows; see OpenAPI for full shape.

## `PUT /v1/update-digital-human/{digital_human_id}`

### Effective tags

- If the body **includes `tag`**, rules use **that** tag.
- Otherwise rules use **existing** stored tags. Changing to a workflow tag in the same request is covered.

### When effective tags are workflow (see above)

**400** if the client **changes** any of these relative to what is stored (value differs from DB):

- **intent** (vs stored description)
- **success_criteria**
- **role_description**
- **original_transcript** or **formatted_transcript**

**Idempotent:** sending the **same** values as stored (e.g. a full PUT round-trip) is allowed.

**`enriched_playback`** remains **updatable** for workflow-tagged digital humans — it is **not** blocked. When `enriched_playback` is not `null`, it is written to the DB.

## `POST /v1/generate-objectives` and `POST /v1/generate-intent-summary`

If the test case’s tags match the workflow rule (§1), the API responds **400** with a message that workflow-tagged digital humans use **enriched playback** on the agent and **objectives / intent summary** are not applicable (wording may vary slightly by endpoint).

## `POST /v1/generate-digital-humans`

For **each** scenario when building the test case:

| Condition | Effect |
|-----------|--------|
| **`enriched_playback`** is a non-empty list | **`description`** and **`success_criteria`** forced to **`""`**. |
| Tags match §1 (workflow) | Same clearing of **description** / **success_criteria**, and **`role_description`** forced to **`null`** (even without `enriched_playback` — e.g. legacy v1 workflow adherence scenarios). |

The batch **objectives / intent summary** step runs only for created rows with **non-empty description** — workflow-style rows with empty intent are **skipped**.

### Workflows v2 scenarios

`auto_gen_workflow_v2_adherence_scenarios` produces scenarios with **empty** `description` / `success_criteria` and **filled** `enriched_playback`. Those feed **`generate-digital-humans`** when **`workflow_adherence_v2`** is used. For graph CRUD and validation, see [Create workflow (v2)](/api-reference/endpoint/create-workflow-v2).
